3   Optimisation des requêtes (4 points)

Sous Oracle SQL on exécute les commandes SQL suivantes:

CREATE TABLE Departement (          CREATE TABLE Compagnie (         
        IdD NUMBER(4),                      NomC VARCHAR(32),        
        NomD VARCHAR(32),                   Siege VARCHAR(32),       
        NomC VARCHAR(32)                    Nbemployes NUMBER(4),    
);                                          President VARCHAR(32)    
                                    );                               

CREATE TABLE Employe (              EXPLAIN PLAN 
        IdE NUMBER(4),                  SET statement_id = 'p0' 
        NomE VARCHAR(32),               FOR 
        DateE VARCHAR(32),          SELECT President 
        IdD NUMBER(4)                 FROM Employe,Compagnie,Departement 
);                                   WHERE Employe.IdD = Departement.IdD 
                                       AND Departement.NomC = Compagnie.NomC 
                                       AND NomE = 'Dupond';
On a créé 3 tables. Les employés ont un identificateur (IdE), un nom (NomE) une date d'embauche (DateE) et un numéro de département (IdD). Un département a un numéro (IdD), un nom (NomD) et un nom de Compagnie (NomC). Une compagnie a un nom (NomC), un siège (Siege) un nombre d'employés (NbEmployes) et un président (Président). On a créé des index qu'on ne montre pas ici.

On lance l'outil EXPLAIN sur une requête SQL
Question 3.1 :
(1/2 point) Que calcule cette requête ?

Solution : Le nom du président des compagnies ou sont employés des Dupond.

Question 3.2 :
(1 point) Le résultat d'EXPLAIN est le plan d'exécution suivant:

Plan d'execution
--------------------------------------------------------------------------------
0 SELECT STATEMENT
  1 MERGE JOIN
    2 SORT JOIN
      3 MERGE JOIN
        4 SORT JOIN
          5 TABLE ACCESS FULL DEPARTEMENT
        6 SORT JOIN
          7 TABLE ACCESS FULL COMPAGNIE
    8 SORT JOIN
      9 TABLE ACCESS FULL EMPLOYE
Expliquez le plan en détail (accès aux tables et index, algorithmes de jointure).

Solution : Tri-fusion pour les 2 jointures. On accède d'abord a la table Employé, on sélectionne au passage les Dupond. On les trie sur le numéro de département et on fait la jointure (fusion) avec le résultat trié sur le numéro de département de la jointure entre Compagnie et Departement (également par tri-fusion, tri des 2 tables sur NomC)

Question 3.3 :
(1/2 point) Qu'en déduisez-vous sur l'existence d'index ?

Solution : Qu'il n'y a pas d'index sur les attributs de jointure
Question 3.4 :
(2 points) On fait les créations d'index suivantes:

CREATE unique INDEX Departement_idd_idx on Departement(IdD);
CREATE INDEX Employe_idd_idx on Employe(IdD);
CREATE unique INDEX Compagnie_nomc_idx on Compagnie(NomC);
CREATE INDEX Employe_nomc_idx on Employe(NomC);
Donnez un plan d'exécution sous forme d'arbre ou sous le format EXPLAIN.

Solution : 
0 SELECT STATEMENT
  1 NESTED LOOPS

       5 INDEX UNIQUE SCAN COMPAGNIE_NOMC_IDX
    6 TABLE ACCESS BY INDEX ROWID EMPLOYE
      7 INDEX RANGE SCAN EMPLOYE_IDD_IDX
